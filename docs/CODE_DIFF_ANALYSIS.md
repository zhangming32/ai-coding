## 2. 详细差异分析

### 2.1 User.java (数据模型)

**差异类型**: 重大数据模型缺失  
**严重程度**: 高

| 字段 | 原始代码 | 生成代码 | 影响 |
|------|---------|---------|------|
| email | 存在 | 缺失 | 用户无法设置邮箱 |
| username | 存在 | 缺失 | 显示名称缺失 |
| role | 存在 | 缺失 | 角色区分依赖ID格式 |
| status | 存在 | 缺失 | 用户状态管理缺失 |
| regTime | 存在 | 缺失 | 注册时间无法追踪 |
| regIp | 存在 | 缺失 | 注册来源无法追踪 |

**SDD规范问题分析**:
- 规范文档 `auth/spec.md` 未明确定义 User 实体的完整属性
- 只描述了登录场景需要的字段，遗漏了用户管理的扩展属性
- **规范改进建议**: 添加 `user` 规范域，完整定义用户实体属性

### 2.2 Page.java (分页模型)

**差异类型**: 数据结构变更  
**严重程度**: 中

| 属性 | 原始代码 | 生成代码 |
|------|---------|---------|
| pageSize | 6 | 10 |
| star (起始位置) | 存在 | 缺失 |
| totalNum (总记录数) | 缺失 | 存在 |
| 泛型支持 | 无 | Page<T> |

**SDD规范问题分析**:
- `system/spec.md` 中定义了分页元数据要求，但具体值未明确
- "Page包含pageSize(每页大小)" 未指定具体数值
- **规范改进建议**: 分页参数应明确定义默认值

### 2.3 Course_limit.java (关联模型)

**差异类型**: 数据库主键缺失  
**严重程度**: 中

原始代码无主键字段，生成代码新增 `limitId` 主键。

**SDD规范问题分析**:
- `course/spec.md` 未明确 Course_limit 表的主键定义
- 场景描述 "创建Course_limit记录" 未说明主键策略
- **规范改进建议**: 数据模型应在 SDD 中明确定义，包括主键

### 2.5 CourseServiceImpl.java (业务逻辑)

**差异类型**: 核心业务逻辑差异  
**严重程度**: 严重

#### 2.5.1 checkStuIns() 方法逻辑反转

```java
// 原始代码：无限制返回 false (不允许选课)
if (limitList == null || limitList.size() == 0) {
    return false;
}

// 生成代码：无限制返回 true (允许选课)
if (limitList == null || limitList.size() == 0) {
    return true;
}
```

**影响**: 无学院限制的课程选课行为完全相反！

**SDD规范问题分析**:
- `course/spec.md` 中的场景描述存在歧义：
  > "无学院限制: GIVEN 课程未设置学院限制 WHEN 任何学生请求选课 THEN 允许选课"
- 规范描述是正确的，但 AI 理解时可能受到上下文影响
- **规范改进建议**: 使用更明确的条件描述，避免歧义

#### 2.5.2 queryStuByCourseId() 缺少成绩设置

```java
// 原始代码
student.setTempScore(course_choose.getScore());

// 生成代码：缺失此行
```

**影响**: 教师查看选课学生时无法看到成绩！

**SDD规范问题分析**:
- `teacher/spec.md` 场景描述：
  > "显示学生学号、姓名、成绩"
- 但未明确说明成绩数据来源和设置方式
- **规范改进建议**: 数据字段映射应在 SDD 中详细说明

#### 2.5.3 queryStuCourse() 缺少成绩字段

生成代码未设置课程对象的 score 字段，导致学生查看已选课程时无法看到成绩。

#### 2.5.4 insertCourse() 缺少初始化

```java
// 原始代码
course.setClassChooseNum(0);

// 生成代码：缺失此行
```

**影响**: 新建课程的已选人数可能为 null。

**SDD规范问题分析**:
- `teacher/spec.md` 场景描述：
  > "初始已选人数为0"
- 规范描述正确，但代码实现遗漏
- **规范改进建议**: 实现验证应对照场景逐一检查

## 4. 代码->规范->代码 转换链路分析

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│  原始代码        │ ──> │  OpenSpec规范    │ ──> │  生成代码        │
│  Course-System  │     │  文档           │     │  ai-coding      │
└─────────────────┘     └─────────────────┘     └─────────────────┘
        │                       │                       │
        ▼                       ▼                       ▼
    实际实现               文档描述                  AI理解实现
        │                       │                       │
        └───────────────────────┴───────────────────────┘
                                │
                          信息丢失点
```

### 4.1 第一阶段：代码 -> 规范文档

**信息保留率**: 约 85%

| 保留信息 | 丢失信息 |
|---------|---------|
| 核心业务流程 | 实现细节（如初始化值） |
| 数据结构定义 | DAO 方法签名 |
| 接口定义 | 内部变量名 |
| 场景描述 | 边界条件处理 |
| 架构设计 | 具体代码实现 |

**丢失原因**:
1. 规范文档侧重描述 "做什么"，不描述 "怎么做"
2. SDD 规范不要求记录所有实现细节
3. OpenSpec 格式限制导致部分信息无法表达

### 4.2 第二阶段：规范文档 -> 生成代码

**信息保留率**: 约 70%

| AI 理解正确 | AI 理解偏差 |
|------------|------------|
| 主要业务流程 | 边界条件处理 |
| 数据模型结构 | 默认值设置 |
| 接口定义 | 方法内部实现 |
| 场景描述 | 数据字段映射 |

**偏差原因**:
1. AI 对歧义描述做出不同解读
2. 规范未覆盖所有边界条件
3. AI 缺乏业务上下文理解

## 6. 结论

### 6.1 关键发现

1. **规范文档信息丢失**: 代码转规范时丢失约 15% 的实现细节信息
2. **AI理解偏差**: 规范转代码时，AI 对歧义描述产生错误解读
3. **最严重问题**: `checkStuIns()` 方法逻辑反转，导致核心业务功能错误
4. **正向改进**: 生成代码修正了原始代码的注解错误和调试代码

### 6.2 规范完善优先级

| 优先级 | 改进项 | 影响 |
|--------|--------|------|
| P0 | 消除场景描述歧义 | 避免逻辑反转 |
| P0 | 增加 DAO 接口定义 | 确保接口兼容 |
| P1 | 增加数据字段映射 | 避免字段遗漏 |
| P1 | 增加默认值定义 | 确保初始化正确 |
| P2 | 增加实体完整属性 | 数据模型完整 |
